  **Step 1 — Install dependencies:**

  npm install zod

  **Step 2 — Create `src/admin/schemas.ts`:**
  Define all 9 Zod schemas mirroring `src/types/data.ts` exactly. Use Zod v4 API (`z.string().min(1)` for required strings, `z.array()` for arrays, `z.boolean()` for booleans). Export sub-schemas too (socialLinkSchema, projectLinkSchema) for reuse in StructuredArrayField validation. Reference the RESEARCH.md code example closely — it has the correct Zod v4 patterns. For NavItem, use `z.object` with `children` as `z.array(z.object({ label: z.string().min(1), href: z.string().min(1) })).optional()` (only one level of nesting per RESEARCH.md recommendation).

  **Step 3 — Create `src/admin/useContentEditor.ts`:**
  Implement the shared hook following the RESEARCH.md Pattern 1 exactly:
  - Generic `useContentEditor<T>({ contentType, schema, onDirtyChange })` signature
  - `useState<T | null>(null)` for data, `useState<FieldErrors>({})` for errors, `useState(true)` for loading
  - `useEffect` fetches `/__admin-api/content/${contentType}` on mount (and when contentType changes)
  - `updateField<K extends keyof T>(field, value)` spreads into data, calls `onDirtyChange(true)`
  - `save()` clears errors (clean slate), runs `schema.safeParse(data)`, on failure: `z.flattenError(result.error)` into fieldErrors + `toast.error('Validation failed -- check highlighted fields')` + return false. On success: POST to `/__admin-api/content/${contentType}`, on 200: `toast.success('Saved')` + `onDirtyChange(false)` + return true. On error: `toast.error(err.error ?? 'Save failed')` + return false.
  - Return `{ data, setData, updateField, fieldErrors, save, loading }`
  - IMPORTANT: Include `data` in the `save` useCallback dependency array to avoid stale closure (Pitfall 2 from RESEARCH.md).
  - The hook also accepts an optional `saveRef: React.MutableRefObject<(() => Promise<boolean>) | null>` and assigns `saveRef.current = save` via a `useEffect` so the AdminShell can trigger saves.

  **Step 4 — Create `src/admin/editors/shared/FormField.tsx`:**
  Labeled input/textarea with error display. Follow RESEARCH.md example:
  - Props: `{ label, value, onChange, error?: string[], multiline?: boolean, placeholder?: string }`
  - Use shadcn `Input` or `Textarea` based on `multiline` prop
  - Red border (`border-red-500`) and red error text below field when `error` has items
  - Uses shadcn `Label` for accessible labels
  - `aria-invalid` attribute when errors present

  **Step 5 — Create `src/admin/editors/shared/TagInput.tsx`:**
  String array editor rendered as tags with Enter-to-add. Follow RESEARCH.md example:
  - Props: `{ label, tags: string[], onChange: (tags: string[]) => void, error?: string[] }`
  - Internal `inputValue` state for the text input
  - On Enter key with non-empty input: append to tags array, clear input. Call `e.preventDefault()` to prevent form submission.
  - Tags rendered as `span` pills with X button. Clicking X removes that tag (immediate, no confirmation per user decision).
  - Use shadcn `Input` for the text field, `Label` for label, lucide `X` icon for remove button.
  - Tags have subtle gray background (`bg-gray-100 text-gray-700`) with rounded corners.

  **Step 6 — Create `src/admin/editors/shared/StructuredArrayField.tsx`:**
  Array-of-objects editor with mini-form rows and add/remove:
  - Generic props: `{ label, items: T[], onChange: (items: T[]) => void, fields: FieldDef[], error?: string[] }` where `FieldDef = { key: string; label: string; placeholder?: string }`.
  - Each item renders as a row of side-by-side `Input` fields (one per FieldDef) with an X button on the right to remove.
  - "+ Add" button at bottom creates a new empty row (all string fields defaulting to `''`).
  - Remove is immediate on X click (no confirmation, per user decision).
  - Error display below the entire array section.

  **Step 7 — Create `src/admin/editors/shared/SectionHeader.tsx`:**
  Simple visual divider:
  - Props: `{ children: React.ReactNode }`
  - Renders as a subtle heading: `text-xs font-semibold uppercase tracking-wider text-gray-400` with a `border-b border-gray-100 pb-1 mb-3` for visual separation.
  - Lightweight — feels like a dev tool section header, not a marketing heading.
