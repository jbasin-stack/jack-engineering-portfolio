  <files>vite-plugin-admin-api.ts</files>
  <action>
In `configureServer`, change from post-middleware (returning a function) to pre-middleware (registering directly).

Current code (lines 108-265) has this structure:
```
configureServer(server) {
  // printUrls override...
  return () => {
    server.middlewares.use(async (req, res, next) => {
      // all API route handling...
    });
  };
}
```

Change to direct middleware registration — do NOT return a function:
```
configureServer(server) {
  // printUrls override stays the same...

  // Register middleware DIRECTLY (pre-middleware) so it runs
  // BEFORE Vite's SPA fallback intercepts /__admin-api/* routes
  server.middlewares.use(async (req, res, next) => {
    // all API route handling... (identical logic, just not wrapped in returned function)
  });
}
```

The key change is removing `return () => {` and its closing `};` — the middleware body stays identical. The `printUrls` override at the top of `configureServer` remains unchanged.

Why this works: Vite's `configureServer` hook has two middleware registration modes:
1. Direct call to `server.middlewares.use()` inside configureServer = PRE-middleware (runs before Vite internals)
2. Returning a function = POST-middleware (runs after Vite internals including SPA fallback)

Our API routes must run before SPA fallback so they return JSON, not index.html.

<task type="auto">
  <name>Task 2: Add error handling to useContentEditor fetch chain</name>
  <files>src/admin/useContentEditor.ts</files>
  <action>
In the `useEffect` that fetches content data (lines 27-34), add a `.catch()` handler to the promise chain.

Current code:
```ts
useEffect(() => {
  setLoading(true);
  fetch(`/__admin-api/content/${contentType}`)
    .then((res) => res.json())
    .then((json) => {
      setData(json as T);
      setLoading(false);
    });
}, [contentType]);
```

Replace with:
```ts
useEffect(() => {
  setLoading(true);
  fetch(`/__admin-api/content/${contentType}`)
    .then((res) => {
      if (!res.ok) throw new Error(`API returned ${res.status}`);
      return res.json();
    })
    .then((json) => {
      setData(json as T);
      setLoading(false);
    })
    .catch((err) => {
      console.error(`[admin] Failed to load ${contentType}:`, err);
      toast.error(`Failed to load ${contentType} data`);
      setLoading(false);
    });
}, [contentType]);
```

Three improvements:
1. Check `res.ok` before calling `res.json()` — catches non-200 responses (like if middleware fix somehow didn't apply)
2. `.catch()` handler ensures `setLoading(false)` always runs, preventing permanent skeleton state
3. Shows error toast so the user gets feedback instead of a silent hang

Note: `toast` is already imported from 'sonner' at line 3 — no new import needed.
