1. **Create/expand `src/admin/__tests__/codegen.test.ts`** (may already have basic tests from Plan 01 TDD -- expand coverage):

   Test `generateDataFile`:
   - Object type (hero): Produces `import type { HeroData } from '../types/data';` + `export const heroData: HeroData = ...;`
   - Array type (projects): Produces `Project[]` type annotation and array literal
   - Handles special characters in data values (apostrophes like "Maxwell's equations" should be safely serialized by JSON.stringify)
   - Produces trailing newline

   Test `formatAndValidate`:
   - Valid input: Returns formatted string with single quotes (not double)
   - Valid input: Output contains trailing commas
   - Valid input: Output is parseable TypeScript (no syntax errors)
   - Invalid input (`const x: = 5;`): Throws Error with message containing line number
   - Invalid input (empty string with only `import type`): Does NOT throw (valid syntax)
   - Verify `import type` is preserved (not converted to `import`)

   Test round-trip:
   - `formatAndValidate(generateDataFile('HeroData', 'heroData', { name: 'Test', subtitle: 'Sub' }, false))` succeeds and output contains `import type { HeroData }`
   - `formatAndValidate(generateDataFile('SkillGroup', 'skillGroups', [{ domain: 'RF', skills: ['Matching'] }], true))` succeeds and output contains `SkillGroup[]`

2. **Create `src/admin/__tests__/atomic-write.test.ts`**:

   To test the atomic write function, it needs to be importable. If `atomicWrite` is not exported from `vite-plugin-admin-api.ts`, extract it to a shared utility or test it indirectly through the plugin. The preferred approach:
   - If `atomicWrite` is already exported or can be imported, test directly
   - If not, create a separate `src/admin/atomic-write.ts` utility that exports `atomicWrite` and the write queue logic, then import it in both the plugin and tests. Update `vite-plugin-admin-api.ts` to import from this utility.

   Tests:
   - **Basic write**: atomicWrite writes content to a temp file in os.tmpdir(), verify file content matches
   - **Atomic behavior**: Write does NOT leave a `.tmp` file after completion (temp file cleaned up by rename)
   - **Content integrity**: Write a known string, read it back, verify exact match
   - **Concurrent writes**: Fire 5 atomicWrite calls to the same file path with different content (`"v1"` through `"v5"`), await all, verify file contains one of the values (last-write-wins, not corrupted)
   - Use `os.tmpdir()` + unique filenames for test isolation, clean up in `afterEach`

   Note: EPERM retry testing is not feasible in unit tests (requires actual antivirus file locking). The retry logic is tested implicitly by the implementation working on Windows.
