<task type="auto">
  <name>Task 2: Add upload endpoint to Vite plugin with busboy multipart parsing</name>
  <files>vite-plugin-admin-api.ts</files>
  <action>
Extend the existing `vite-plugin-admin-api.ts` with a new upload route. Add to the existing middleware, do not create a separate plugin.

1. Add imports at top of file:
   - `import Busboy from 'busboy';`
   - `import { writeFile, mkdir } from 'fs/promises';`
   - `import { extname, dirname } from 'path';`
   - `import { toKebabCase, validateUpload, getUploadPath } from './src/admin/upload';`

2. Add upload route handler BEFORE the existing `/__admin-api/content/:type` matcher in the middleware:

   Match: `url === '/__admin-api/upload' && req.method === 'POST'`

   Handler logic:
   a. Create Busboy instance with `{ headers: req.headers, limits: { fileSize: 10 * 1024 * 1024 } }`
   b. Track: `fileBuffer` (Buffer | null), `fileName` (string), `fileMimeType` (string), `context` (parsed JSON), `fileTruncated` (boolean for size limit)
   c. On `bb.on('file', ...)`: collect chunks into buffer, capture filename from `info.filename` and mimeType from `info.mimeType`. Listen for `stream.on('limit', ...)` to set fileTruncated=true.
   d. On `bb.on('field', ...)`: parse the `context` field as JSON with `contentType`, `field`, `itemId`
   e. On `bb.on('finish', ...)`:
      - If fileTruncated: respond 413 `{ error: 'File too large (max 10MB)' }`
      - If no fileBuffer or no context: respond 400 `{ error: 'Missing file or context' }`
      - Validate with `validateUpload(fileName, fileBuffer.length)` -- if error, respond 400
      - Normalize filename: `toKebabCase(fileName)`
      - Get extension: `extname(normalizedName)`
      - Compute target path: `getUploadPath(context.contentType, context.field, context.itemId, ext)`
      - Full disk path: `resolve(server.config.root, 'public', targetPath.replace(/^\//, ''))`
      - Ensure directory exists: `await mkdir(dirname(fullPath), { recursive: true })`
      - Write file: `await writeFile(fullPath, fileBuffer)` (binary write, NOT text -- no encoding param)
      - Update data file reference:
        * Load current data via `server.ssrLoadModule('/src/data/' + entry.file)` where entry comes from CONTENT_REGISTRY[context.contentType]
        * Clone the data, update the relevant field with the new path (e.g., for projects, find the item by id and set its thumbnail field)
        * Generate new data file source via `generateDataFile`
        * Format and validate via `formatAndValidate`
        * Write via `enqueueWrite` with the same activeWrites tracking as existing content writes
      - Respond 200 `{ success: true, path: targetPath }`
   f. Pipe request: `req.pipe(bb)`
   g. Handle errors: wrap in try/catch, respond 500 on unexpected errors

   Data reference update logic (inside the finish handler):
   - For `projects`: find item in array by `context.itemId`, update `context.field` (thumbnail or images)
   - For `papers`: find item by `context.itemId`, update `pdfPath`
   - For `hero`/portrait: this is a special case -- the portrait path is not in heroData. Handle by checking if contentType is "hero" and field is "portrait", then just write the file (no data update needed since portrait.jpg path is hardcoded in the component)
   - For `contact`/resume: update `resumePath` field on the singleton object
   - If the content type or field doesn't map to a data update, just write the file and skip data update (respond with success)

   IMPORTANT: Write the file to public/ FIRST, THEN update the data file. This ordering prevents the race condition where HMR fires before the file is written (see Research pitfall 3). Public file writes do NOT trigger HMR (Vite only watches src/).
